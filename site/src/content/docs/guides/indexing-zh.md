## 初始化和索引

```bash
cd your-project
codegraph init -i      # 初始化 + 完整索引
```

`init` 创建 `.codegraph/` 目录；`-i`/`--index` 立即构建索引。要仅初始化而不索引，省略该标志，稍后运行 `codegraph index`。

## 完整索引与增量索引

```bash
codegraph index           # 对整个项目进行完整索引
codegraph index --force   # 从头开始重新索引
codegraph sync            # 增量 — 仅处理已更改的文件
```

`sync` 速度很快，因为它只重新解析已更改的内容。在切换分支或批量编辑后使用它。

## 自动保持最新

**在代理会话期间，你不需要手动运行 `codegraph sync`。** 当你的代理（Claude Code、Cursor、Codex、opencode、Hermes、Gemini、Antigravity、Kiro）启动 `codegraph serve --mcp` 时，三个层级协同工作，使索引与你的代码保持同步 — 并在编辑与下一次同步之间的短暂窗口内，确保代理永远不会得到静默的错误答案。

### 1. 带防抖自动同步的文件监视器（始终开启）

`serve --mcp` 会在项目根目录上启动一个原生文件监视器（macOS 上的 FSEvents、Linux 上的 inotify、Windows 上的 ReadDirectoryChangesW）。每个源文件的创建/修改/删除都会被捕获。一个防抖定时器将批量编辑合并为一次同步。

```
代理写入 src/Widget.ts
  → 监视器触发（事件传递：通常 <100ms）
  → 2000ms 防抖
  → 同步运行；Widget.ts 的节点 + 边已进入索引
  → 下一次代理查询时即可看到
```

**可调参数**：`CODEGRAPH_WATCH_DEBOUNCE_MS` 可覆盖默认的 2000ms，限制范围为 `[100ms, 60s]`。当构建步骤或格式化工具在短时间内写入大量文件时很有用 — 将其增加到 `5000` 或 `10000`，让监视器将它们合并为一次同步。

### 2. 每个文件的过时横幅 — 覆盖防抖窗口

监视器防抖引入了一个小窗口（通常为 2 秒），期间新编辑的文件已存在于磁盘上但尚未进入索引。CodeGraph 通过每个文件的过时横幅来关闭这个窗口：如果某个 MCP 工具响应会引用当前待重新索引的文件，响应会前置一个 `⚠️` 横幅，指出该文件的名称：

```
⚠️ 以下引用的某些文件自上次索引同步以来已被编辑 —
它们的 codegraph 条目可能已过时：
  - src/Widget.ts（800 毫秒前编辑，等待同步）
要获取这些特定文件的准确内容，请直接读取（Read）它们。
响应的其余部分是最新的。

## 代码上下文
…
```

代理会读取这一信息，并通过直接 `Read` 指定文件来跟进 — 已在 Claude Code 上经过端到端验证，代理会在打开文件前明确说"正在直接读取文件以获取最新内容"。因此，即使在 2 秒的防抖窗口内，代理也永远不会得到静默的错误答案。

响应中**未**引用的待处理文件会以小型页脚形式显示（`（注意：本项目中还有 N 个文件正在等待索引同步，但未在上面被引用：…）`）。无论哪种方式，信号都是明确的。

### 3. 连接时追赶 — 覆盖 MCP 服务器未运行时的间隔

当你的编辑器/代理（重新）连接到 MCP 服务器时，codegraph 会在回答第一个查询之前运行一次快速的基于文件系统的对账（先进行 `(size, mtime)` 状态预过滤，然后对剩余文件进行内容哈希检查）。因此，在 MCP 服务器未运行时发生更改的文件 — 来自终端的 `git pull`、来自其他编辑器的编辑、已完成并退出的代理 — 会在下一个会话的第一个工具调用时自动被追赶。

### 验证监视器看到的内容

`codegraph_status` 直接以一等公民的方式暴露待处理集合 — 这对代理通过一次调用询问"索引是否已追赶？"非常有用：

```
codegraph_status →
  ## CodeGraph 状态
  …
  ### 待同步：
  - src/Widget.ts（1200 毫秒前编辑）
```

如果响应中没有 `### 待同步：`，则说明没有进行中的操作。

### 何时需要手动 `codegraph sync`

几乎不需要。边缘情况如下：

- **监视器被禁用。** 阻止本地文件系统监视器的沙箱，或者你设置了 `CODEGRAPH_NO_DAEMON=1` 来选择不使用共享守护进程。在这些情况下，`codegraph sync` 是手动后备方案。
- **CI 运行前的预检查。** 如果你在代理会话之外对索引进行脚本操作，在脚本开始时执行一次 `codegraph sync` 可确保索引反映当前的工作树。

否则：放心使用。监视器 + 横幅 + 连接同步覆盖了 AI 辅助工作流的端到端需求。如果你在防抖窗口过去后发现文件确实被遗漏了，那是一个 bug — 请提供复现步骤并提交 issue。

> 参见 v0.9.5 发布说明了解[过时横幅（#403）](https://github.com/colbymchenry/codegraph/releases/tag/v0.9.5)和连接时追赶（#414）；两者一同发布。

## 检查状态

```bash
codegraph status
```

报告节点/边/文件的数量、活动的 SQLite 后端和日志模式。在代理会话中，MCP 侧的 `codegraph_status` 还会额外显示上述的 `### 待同步：` 块。

## 哪些内容会被索引

每个扩展名映射到[支持的语言](/codegraph/reference/languages/)的文件，减去默认排除的依赖/构建目录（`node_modules`、`vendor`、`dist`、……）、`.gitignore` 排除的任何文件以及超过 1 MB 的文件。请参阅[配置](/codegraph/getting-started/configuration/)。
