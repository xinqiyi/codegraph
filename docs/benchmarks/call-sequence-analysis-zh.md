# 调用序列分析——读取节省为何未转化为墙钟时间节省

**日期：** 2026-05-23 · **分支：** `architectural-improvements` · **数据来源：** A/B 矩阵中保留下来的 stream-json 日志（`/tmp/ab-matrix/<Cell>/run-headless-{with,without}.jsonl`，37 个单元格 × 2 个实验组）。重新挖掘——**未重新运行**——使用 `scripts/agent-eval/seq-matrix.mjs`。

## 为何有此分析

[A/B 矩阵](codegraph-ab-matrix.md) 显示 codegraph 将**读取减少了 75%**，但**墙钟时间仅减少约 16%**，且 63% 的墙钟时间收益仅来自 3 个大型仓库单元格。读取已降到最低（约 0），因此剩余的墙钟时间消耗在**往返次数 + 综合生成轮次**上——这两者都无法用读取次数来解释。矩阵记录了工具的*次数*，但没有记录调用的**序列**或每次调用的**载荷大小**。本分析恢复了这两者，以找出墙钟时间实际花在了哪里。

## TL;DR——瓶颈是 trace 的采用率，而非 trace 的完备性

1. **Trace 在 37 个单元格中仅被调用了 3 次**——尽管每个问题都是经典的流程追踪问题（"追踪 controller → service → repository"，"X 如何到达 Y"）。智能体压倒性地倾向使用 **`context → search → search → explore`** 来代替——这恰恰是指令告诉它要避免的路径重建反模式。
2. **`explore` 平均每次调用 17.9K 字符；`trace` 平均每次调用 0.8K**——载荷相差 **22 倍**。那个解决小型仓库膨胀问题的路径范围工具是存在的，而且很小。它只是没有被调用。
3. **小型仓库仍然获得臃肿的载荷**，因为默认使用了 explore：一个 **6 个文件** 的仓库（`flutter_module_books`）拉取了 **17.4K**；一个 10 个文件的仓库拉取了 18.0K。这正是"小代码库上上下文过多"的故障模式——此刻正在发生，通过 explore。
4. **使用 codegraph 时往返次数减少 25%（283 对比 375 轮）**，但墙钟时间仅快 16%——因为使用 codegraph 的实验组每轮都携带约 18K 的 explore 载荷，增加了首令牌时间（TTFT），侵蚀了轮次节省。
5. **根本原因：** `src/mcp/server-instructions.ts` 以*"直接回答……先使用 `codegraph_context`，然后使用一次 `codegraph_explore`"* 作为标题模式。trace 优先的指导被埋在一个表格及其下方的链式列表中。智能体锚定在突出的标题上 → context→explore。

**决定：** 下一个实验是 **trace 优先的引导/采用率**，而不是丰富 trace 的内容。当 trace 在 37 次中只被使用 3 次时，我们无法评估它的完备性。先提高采用率，再衡量残余的 `node`/`explore` 后续调用是否需要更丰富的 trace。

## 发现 1——trace 采用率：3/37

| 指标 | 值 |
|---|---|
| 流程问题单元格数 | 37（全部） |
| 调用了 `codegraph_trace` 的单元格数 | **3**（`cpp-leveldb`、`excalidraw`、`c-redis`） |
| 替代的主流模式 | `context` → `search`×N → `explore` |

这 3 个 trace 单元格，以及 trace 调用之后的内容：

| 仓库 | 文件数 | cg 调用序列 | 轮次（有/无） |
|---|--:|---|---|
| cpp-leveldb | 134 | `trace, node, node` | 5 / 8 |
| excalidraw | 643 | `context, trace, trace, explore` | 6 / **19** |
| c-redis | 884 | `context, trace, explore, node` | 10 / 15 |

即使 trace *被* 使用了，智能体也会随后调用 `node`/`explore` 来获取主体内容——因此第二个杠杆（在采用率之后）是使一次 trace 调用足够自给自足，以消除那些后续调用。但这是第二步。

## 发现 2——载荷大小：路径范围 trace（0.8K）vs 广度范围 explore（17.9K）

跨所有单元格，按 codegraph 工具统计——调用次数和**每次调用的平均载荷**：

| 工具 | 调用次数 | 平均每次 | 总计 |
|---|--:|--:|--:|
| `explore` | 32 | **17.9K** | 573K |
| `context` | 36 | 4.3K | 156K |
| `search` | 39 | 1.3K | 50K |
| `files` | 5 | 3.4K | 17K |
| `node` | 19 | 2.0K | 38K |
| `trace` | 4 | **0.8K** | 3.4K |

`context`（在 36/37 个单元格中使用）是默认的开场工具；`explore` 是默认的收尾工具。两者一起构成了约 22K 的广度转储。`trace`——那个本可以用实际路径替换它的工具——小了 22 倍，且几乎未被使用。这证实了用户的假设：explore 是广度范围的（返回邻近区域），trace 是路径范围的（返回线路）。

## 发现 3——载荷随仓库大小增长，并在小型仓库上过度返回

使用 codegraph 实验组 **总计** codegraph 载荷按仓库规模分层：

| 层级 | 单元格数 | 平均总载荷 | 范围 |
|---|--:|--:|--:|
| S（<200 个文件） | 19 | 12.7K | 3.0–31.2K |
| M（<2000） | 9 | 32.4K | 5.4–58.2K |
| L（≥2000） | 9 | 34.0K | 20.2–43.1K |

小型仓库的浪费是具体的——这些仓库都有一个 2–3 文件的流程，但拉取了完整的邻近区域：

| 仓库 | 文件数 | 使用 cg 载荷 | 调用序列 |
|---|--:|--:|---|
| flutter_module_books | 6 | 17.4K | `context, explore` |
| computer-database | 10 | 18.0K | `context, search, status, explore` |
| aspnet-realworld | 78 | 22.2K | `context, explore` |
| django-realworld | 44 | 14.8K | `context, explore` |

`explore` 的每次调用预算已经是自适应的（#185），但在这里无济于事，因为智能体没有选择路径范围工具——它选择的是广度。

## 发现 4——往返次数和 ToolSearch 开销

| 指标 | 使用 codegraph | 未使用 |
|---|--:|--:|
| 总轮次（37 个单元格） | 283 | 375 |
| 平均每单元格轮次 | 7.6 | 10.1 |

轮次减少 25%，但墙钟时间仅快约 16%——差距在于大型 explore 载荷带来的每轮成本。另外：**每次使用 codegraph 的运行都以一次 `ToolSearch` 往返开始**（在此测试工具中 MCP 工具是延迟加载的），这是在任何 codegraph 调用之前的一个固定 1 轮开销。值得确认生产环境安装是否也会以相同方式延迟加载 codegraph 工具。

## 结论 → 接下来要运行的实验

先测量后行动改变了计划。原先的假设是"丰富 trace 使其一次调用即可自给自足"。数据表明 trace **在 37 次中只被使用了 3 次**，因此在采用率问题解决之前，完备性无从谈起。

**实验：trace 优先引导 A/B。**
- **变更：** 重写 `server-instructions.ts` 的标题，使得*流程*问题（X 如何到达 Y / 追踪 / 从→到）**首先**路由到 `codegraph_trace`，将 context→explore 模式降级用于非流程/上手问题。同步更新到 `instructions-template.ts` + `.cursor/rules/codegraph.mdc`。
- **指标：** trace 采用率（目标远大于 3/37），使用 codegraph 实验组总载荷（预期大幅下降，尤其是小型仓库），轮次（预期下降），墙钟时间（预期 16% 的差距会向 25% 的轮次差距靠拢，因为 18K 的 explore 载荷被 <1K 的 trace 替代）。
- **对照：** 一个非流程问题"模块 X 是做什么的"仍然应该走 context→explore——不要把所有东西都过度引导到 trace。
- **然后，第二步：** 采用率提高后，测量 trace 之后的 `node`/`explore` 后续调用（cpp-leveldb/excalidraw/c-redis 都有）。如果它们很频繁，则丰富 trace（每跳主体片段，每跳有上限），使一次 trace 调用就能结束流程调查。

# 消融实验——`context`、`explore` 和 `trace` 是否相互竞争？`trace` 是否足够？

**日期：** 2026-05-23 · 52 次运行，约 $20。工具面在**服务端**通过新的 `CODEGRAPH_MCP_TOOLS` 白名单进行裁剪（因此被消融的工具在 ListTools 中真正消失，而非在调用时被拒绝）；trace 优先引导通过 `--append-system-prompt` 注入。6 个仓库（2 个小型 / 2 个中型 / 2 个大型）× 2 次运行；实验组 E 是在 2 个仓库上的**非流程**调查问题。驱动脚本 `arms-matrix.sh`，分析 `parse-arms.mjs`。

| 实验组 | 工具 | 引导 | 采用率 | 读取 | cgOut | 轮次 | 时长 |
|---|---|---|---|---|--:|--:|--:|--:|--:|
| **A** 对照 | 全部 | 无 | 2/12 | 1.25 | 28.8K | 7.6 | 38s |
| **B** 引导 | 全部 | trace 优先 | **8/12** | 1.00 | **32.0K** | 7.9 | 43s |
| **C** 无 explore | 隐藏 explore | trace 优先 | 8/12 | **2.08** | **9.2K** | 9.0 | 44s |
| **D** trace 中心 | 隐藏 explore+context | trace 优先 | 8/12 | 2.00 | 6.6K | 10.5 | 46s |
| **E** 对照探测 | 隐藏 explore+context | trace 优先 | 0/4 | 2.50 | 27.8K | **20.0** | **72s** |

## 结果分析

1. **引导对采用率有效，但对载荷无效。** B 将 trace 使用从 **2/12 提升至 8/12**（并且在真正路径形状的问题上达到 4/4——那 2 个未采用的，flutter 的"哪些 widgets"和 vapor 的"命名路由"，不是 from→to 问题）。但 B 的载荷（32.0K）*大于*对照（28.8K）且稍慢——因为智能体调用了 trace **并且仍然调用了 explore**。引导增加了一次 trace 轮次，但没有取代 explore 转储。
2. **`explore` 是载荷的元凶，并且它是承载性的——但过重了 3–5 倍。** 移除它（C）削减了 **71%** 的载荷（32K→9.2K）——确认了它是膨胀源。但读取**翻倍**（1.0→2.1），轮次增加：智能体通过 Read 文件来恢复 explore 原先内联的主体内容。因此 explore 并非冗余；它是唯一的一次调用主体供应者，只是以 32K 的大锤方式交付。
3. **`context` 是三者中最冗余的——作为主体供应者而言。** 在 explore 之上再移除它（D vs C）使读取保持平稳（2.08→2.00）但增加了轮次（9.0→10.5）。它不提供任何独特的主体内容；它作为轮次节省者（组合式定向调用）才保住了自己的位置。
4. **移除工具使流程问题变慢，而非变快。** 轮次单调递增 A→D（7.6→10.5），时长也随之增加——Read + trace 后续调用的往返消耗了比节省的载荷更多的墙钟时间。更精简的载荷 ≠ 更快。
5. **`trace` 明确来说是不够的。** 非流程探测（E）在没有调查工具的情况下表现糟糕——**20 轮，72s**，通过 search/node/files 重建概览。调查问题需要一个调查工具；trace 无法替代。

## 关于三个设计问题的结论

- **我们需要全部三个吗？** 是的——但出于不同的原因。trace = 流程工具（真实需求，采用不足）。explore = 一次调用主体供应者（承载性，过重）。context = 节省轮次的开场工具（对主体内容而言冗余，对方向定位有用）。
- **它们是否相互竞争？** 是的：explore 与 trace 竞争并且**默认胜出**——即使经过引导，智能体仍然会 trace **并且** explore，因此在探索被取代之前，载荷优势永远不会实现。
- **trace 能否成为我们唯一需要的工具？** 不能。E 排除了它在非流程问题上的可能性；C/D 排除了它即使在流程问题上的可能性（没有 explore 的主体内容，读取翻倍）。

**现在数据排除了三种廉价的修复方案：** "trace 就是一切"（错误）、"只需引导到 trace"（B：比对照更慢且更大）、以及"移除 explore"（C/D：更多读取/轮次，更慢）。

## 数据指向的修复方案 → 下一个实验

唯一胜出的路径：**使 `trace` 自给自足，通过内联每跳的主体内容**（每跳有上限 → 仍然是路径范围），使得一次 trace 调用能提供 explore 所能提供的*以及* Read 后备所能恢复的内容——从而在流程问题中取代两者。保留**一个**调查工具（context；将 explore 降级为深度调查，而非流程默认）用于非流程类问题——E 已证明此类问题是承载性的。

- **实验：** 丰富的主体内联 `trace` + 引导 vs 对照。
- **目标：** C/D 的精简载荷（约 7–9K，而非 32K）**且没有** C/D 的额外读取/轮次，并**在墙钟时间上超越 A**（B/C/D 都未能达到的基准）。
- **指标：** 载荷、读取（必须保持约等于 A 的约 1.0，而非上升到 2.0）、轮次、时长。

# 验证——主体内联 trace（实验组 F）

消融实验指向了一个修复方案：使 `trace` 自给自足，通过内联每跳的**主体内容**（每跳有上限 → 仍然是路径范围），使得一次 trace 调用既能取代 explore 转储，也能取代 Read 后备。在 `handleTrace` 中实现（`sourceRangeAt`，每跳 28 行 / 1200 字符，带有 `…（+N 行更多）` 标记）。实验组 **F** = 实验组 B 的工具面（全部工具 + trace 优先引导）在主体内联构建上运行，因此 **F vs B 隔离了丰富化的效果**。

| 实验组 | 采用率 | 读取 | cgOut | 轮次 | 时长 | 成本 |
|---|--:|--:|--:|--:|--:|--:|
| A 全部/无引导 | 2/12 | 1.25 | 28.8K | 7.6 | 38s | $0.390 |
| B 全部/引导（瘦 trace） | 8/12 | 1.00 | 32.0K | 7.9 | 43s | $0.411 |
| **F 全部/引导（主体 trace）** | 5/12 | **1.17** | **25.1K** | **6.8** | **37s** | **$0.348** |
| C 无 explore | 8/12 | 2.08 | 9.2K | 9.0 | 44s | $0.356 |
| D trace 中心 | 8/12 | 2.00 | 6.6K | 10.5 | 46s | $0.368 |

**F 是最平衡的实验组：** 轮次最少（6.8），最快（37s），最便宜，载荷比 A/B 更精简——并且达到了消融实验设定的目标：**C/D 级别的效率，但没有 C/D 的 Read 惩罚**（F 读取 1.17 vs C/D 的约 2.0）。它实现这一点不是通过*移除*一个工具，而是通过给智能体一个完整的 trace，让它*提前停止*。

**在 trace 能够连接的地方，优势最为明显**——excalidraw（经过验证的 6 跳路径）：

| 实验组 | 调用序列 | 轮次 | 读取 | 时长 |
|---|---|---|---|--:|--:|--:|
| B（瘦） | `trace → context → explore → Grep → Read` | 7 | 1 | 47s |
| **F（主体）r1** | `trace → context` | **4** | **0** | **31s** |
| F（主体）r2 | `trace → trace → explore` | 5 | 0 | 42s |

主体 trace 在 `trace → context`（运行 1）时就结束了调查——0 次读取，0 次 grep，0 次 explore。

**连通性是上限。** 在因*未桥接*的动态分发而断裂的流程上——aspnet-realworld（MediatR `_mediator.Send → Handle`）、vapor-spi（闭包路由）——trace 返回"无路径"，智能体回退到 explore，因此 F ≈ B（无回归，无增益）。F 的总体提升因此**受限于动态分发覆盖度**：图形端到端连接的流程越多，自给自足 trace 触发的频率就越高。（n=2/实验组——采用率和按仓库的数字有噪音；excalidraw 和 spring-halo，即那些能连接的仓库，在 B 和 F 中都是 2/2 的 trace 采用。）

## 结论与发布清单

1. **发布主体内联 trace**——严格的改进（最平衡的实验组；在能连接的 trace 上实现了干净的 0 读取/4 轮次胜利；在不能连接的 trace 上无回归）。
2. **加强引导。** 实验组 A（已发布的 server-instructions，它*已经*说"流程问题先 trace"）的 trace 采用率只有 2/12——指导信息被埋得太深了。B–F 中使用的显式 `--append-system-prompt` 提升了它。将其移植到 `server-instructions.ts` + `instructions-template.ts` + `.cursor/rules/codegraph.mdc`（内部规则：三者一起更新），按流程类型做门控，使得非流程调查问题仍然走 context/explore（实验组 E 证明了它们必须这样做）。
3. **扩大 F 覆盖范围的下一个前沿：** 桥接更多动态分发（MediatR/.NET、Vapor 路由）——每个新连接的流程都会将一个 F≈B 的仓库转化为 F 胜出的仓库。

# 引导移植——负面结果（实验组 G）

F 的胜利使用了 `--append-system-prompt`，而真实用户得不到这个。实验组 **G** = 实验组 A 的调用方式（无附加提示）在一个将引导移植到生产渠道（`server-instructions.ts` + `context`/`trace` 工具描述 + `instructions-template.ts` + `.cursor/rules`）的构建上运行。三种措辞迭代，每次 12 次运行：

| 实验组 | 采用率 | 读取 | 载荷 | 轮次 | 时长 |
|---|--:|--:|--:|--:|--:|
| A（已发布指令） | 2/12 | 1.25 | 28.8K | 7.6 | **38s** |
| F（主体 trace + 附加提示） | 5/12 | **1.17** | 25.1K | 6.8 | **37s** |
| G v1 — 反 explore 措辞 | 6/12 | 2.08 | 13.8K | 8.8 | 46s |
| G v2 — 恢复 explore 作为后备 | 6/12 | 1.67 | 22.0K | 7.8 | 46s |
| G v3 — 恢复 context 作为开场 | 6/12 | 2.08 | 11.7K | 8.9 | 46s |

**生产环境指令引导无法复现 F，并且使 A 的基线倒退。** 所有三个 G 变体都固定在 **约 46s**（比 A 的 38s 和 F 的 37s 更慢），读取在 1.7–2.1 之间（vs A 1.25，F 1.17）。措辞只在 Read 和 explore 之间重新分配了 slack——v1 抑制了 explore → Read 增加；v2/v3 恢复了 explore → 过度调查——从未实现 F 的精简 `trace → context`。

**两个根本原因：**
1. **显著性。** 相同的 trace 优先措辞作为提示开头的 `--append-system-prompt`（F）有效，但作为 MCP `initialize` 指令/工具描述（G）无效。MCP 服务器没有更高显著性的渠道——这是一个架构限制，而非措辞错误。
2. **强制 trace 优先在 trace 无法连接的地方适得其反。** 引导将 trace 推向了 MediatR（`_mediator.Send`）和 Spring 接口 DI（`@Autowired` 接口 → 实现）的流程，在这些流程中 trace 返回无路径；强制 trace 然后是一个浪费的往返*在*回退之前→更慢。**未引导**的智能体（A）校准得更好：它只在 trace 明显能连接时（2/12）使用 trace，其他情况则使用 explore。

## 实验组 H——仅主体 trace（待发布候选）出现倒退

干净的发布测试：主体内联 trace + 原始指令 + 无引导（= A 的调用方式，只有 trace *工具*改变了）。H vs A 隔离了主体 trace 功能，其他因素不变。

| 实验组 | 采用率 | 读取 | 载荷 | 轮次 | 时长 |
|---|--:|--:|--:|--:|--:|
| A（无主体 trace） | 2/12 | 1.25 | 28.8K | 7.6 | **38s** |
| H（主体 trace，无引导） | 3/12 | 1.50 | 29.7K | 8.0 | **45s** |
| F（主体 trace + 附加提示） | 5/12 | 1.17 | 25.1K | 6.8 | 37s |

**单独的主体 trace 并没有击败 A——它轻微倒退**（45s vs 38s）。调用序列揭示了原因：未引导时，智能体将 trace 视为其常规循环中的又一个调用——excalidraw H 的序列是 `context → trace → explore → node×3 → Grep → Read`（77s）——因此更大的主体 trace 载荷纯粹是额外成本，没有被更少的后续调用所抵消。只有当智能体**以 trace 开头并在之后停止**时，主体 trace 才有回报——而这只有附加提示（F）才实现了。

## 最终结论

主体内联 trace 是一个真正的胜利（F），但它的价值**完全依赖于"以 trace 开头并在之后停止"的引导，而我们无法通过任何生产 MCP 渠道提供这种引导**（附加提示的显著性远高于 server-instructions / 工具描述；G 失败了三次）。单独使用时（H）它出现倒退。因此：

- **发布：`CODEGRAPH_MCP_TOOLS` 白名单**——独立、干净、已验证。
- **不要按原样发布主体内联 trace 或引导**——在没有我们不具备的引导渠道的情况下，测量结果为中性到负面。
- **真正的杠杆是连通性，而非引导**——trace 只有在流程端到端连接时才能体现其价值；动态分发合成器（MediatR/.NET、Spring 接口 DI、Vapor 闭包）能够帮助*未引导的*智能体，它已经在 trace 可能连接时使用 trace。
- **一个未测试的杠杆**可以挽救主体 trace：通过 trace 工具*自身的输出*进行引导（最高显著性的渠道——智能体在决策点新鲜地读取它），带有一个强烈的开场"完整流程——据此回答，不要 explore"。指令/描述离行动太远；工具结果则不是。未经证实；这是在生产中使主体 trace 获得回报的唯一剩余机会。

先测量后行动三次带来了回报：它在消融实验中排除了三种廉价修复方案，阻止了一个会导致每次查询约 8s 回归的引导变更（G），并且阻止了在混淆假设下发布主体 trace 本身（H 表明它需要我们无法提供的引导）。

## 复现（实验组 G）

```bash
ARM=G bash scripts/agent-eval/arms-F.sh    # 生产环境指令引导，无附加提示
node scripts/agent-eval/parse-arms.mjs
```

# 实验组 I——充分性，而非引导（可发布的胜利）

LLM 在其上下文*足够*时停止调查，而不是被告知停止时。因此实验组 I 使 trace 的**输出**完备而非引导——与 H 相同的调用方式（原始指令，**无引导**），只有 trace 工具改变了：
1. **跳主体不再被截断**到 28 行（正是这个截断导致 H 重新获取了 `mutateElement`）。
2. **目标的自身被调用者被内联**——智能体否则会使用 explore/Read 获取的"最后一英里"（excalidraw：`renderStaticScene → _renderStaticScene / renderStaticSceneThrottled`）。

| 实验组 | 采用率 | 读取 | greps | 载荷 | 轮次 | 时长 | 成本 |
|---|--:|--:|--:|--:|--:|--:|--:|
| A 基线 | 2/12 | 1.25 | 1.17 | 28.8K | 7.6 | 38s | $0.390 |
| H 仅主体 trace | 3/12 | 1.50 | 0.42 | 29.7K | 8.0 | 45s | $0.398 |
| **I 主体 trace + 目标被调用者** | 2/12 | **1.17** | **0.25** | 27.2K | **7.0** | 39s | **$0.359** |
| F 主体 trace + 附加引导 | 5/12 | 1.17 | 0.17 | 25.1K | 6.8 | 37s | $0.348 |

**I 在每个维度上 ≥ A**（读取、grep、轮次、成本下降；墙钟时间持平）并且**在零引导的情况下 ≈ F 的结果**——尽管 trace 采用率*更低*（2/12 vs F 的 5/12）。目标被调用者的修复将主体 trace 从净负面（H，45s）转变为净正面（I，39s）：一次更丰富的 trace 调用现在取代了它过去触发的 explore+node+Read 后续调用。excalidraw I-r2 的序列是 `context → trace → explore`——**0 次读取，5 轮**，因为数据已经存在就停止了。残余的读取（I-r1）是 `canvasNonce` 的数据流——图故意省略的 def-use 前沿。

这证实了论点：**完备性使智能体停止；引导不能。** 每一个引导实验组（B/F 附加提示，G 指令）要么不可发布，要么是回归；充分性实验组（I）可以发布，且不需要任何引导。

## 修订后的最终结论（取代上述实验组 G/H 的结论）

- **发布：主体内联 trace + 目标被调用者**（实验组 I）——在所有维度上 ≥ A，无需引导，无回归；使自给自足 trace 属性成为现实（一次 trace 调用回答流程问题）。
- **发布：`CODEGRAPH_MCP_TOOLS` 白名单**——独立、已验证。
- **不要发布引导**（指令或工具描述）——三个变体均出现倒退；MCP 无法提供附加提示的显著性，且在 trace 无法连接时强制使用 trace 会适得其反。
- **连通性是倍增器**——实验组 I 在 trace 能连接的地方帮助最大；MediatR/.NET、Spring 接口 DI 和 Vapor 闭包是下一个合成器，它们能帮助*未引导的*智能体（它已经在 trace 可能连接时使用 trace）。

## 复现（实验组 I）

```bash
ARM=I bash scripts/agent-eval/arms-F.sh    # 主体 trace + 目标被调用者，无引导
node scripts/agent-eval/parse-arms.mjs
```

# 当前构建的有/无 A/B——7 个 README 仓库（2026-05-24）

重新运行了已发布 README 基准测试，使用**当前构建**（全部 7 个仓库重新索引），相同查询，**每次运行 4 次的中间值**（无头模式：仅 codegraph MCP vs 空 MCP）：

| 仓库 | 有→无时间 | 工具有→无 | 令牌有→无（节省） | 成本有→无（节省） |
|---|---|---|---|--:|--:|--:|
| vscode | 1m10s→2m26s | 8→55 | 601k→2.8M（78%） | $0.60→$0.80（26%） |
| excalidraw | 48s→2m58s | 3→79 | 344k→3.5M（90%） | $0.43→$0.90（52%） |
| django | 1m19s→1m38s | 9→19 | 739k→1.2M（36%） | $0.59→$0.67（12%） |
| tokio | 53s→3m2s | 4→53 | 379k→2.6M（86%） | $0.42→$2.41（82%） |
| okhttp | 42s→1m1s | 6→11 | 636k→730k（13%） | $0.47→$0.47（2%） |
| gin | 44s→1m0s | 6→10 | 444k→675k（34%） | $0.37→$0.47（21%） |
| alamofire | 1m17s→2m27s | 12→69 | 1.0M→2.8M（64%） | $0.61→$1.14（47%） |

**平均节省：35% 成本 · 57% 令牌 · 46% 时间 · 71% 工具调用**——复现了已发布的 README 标题数据（35% / 59% / 49% / 70%）；当前构建保持了基准，无回归。

**成本更低，而非"持平"**（修正了之前的注释）。但**机制是体积，而非缓存可命中性**：codegraph 在更少的轮次中以更小的累计上下文回答问题，而无 codegraph 实验组则在更多轮次中扩散（大仓库上 55–79 次工具调用），每轮重新处理一个庞大且不断增长的上下文。无 codegraph 实验组的令牌量*主要是*廉价的缓存读取，这就是为什么**令牌数节省（57%）看起来比成本节省（35%）大**。按仓库的边际差异取决于无 codegraph 实验组在该次运行中的挣扎程度（tokio 膨胀到 $2.41/3m；django 挣扎程度较低）。

**测量陷阱：** 这个 Claude Code 版本中的 `result.usage` 是**仅最后一轮**的，而非累计值——使用它会严重低估令牌数（较早的一次 excalidraw 数据报告了基于这个 bug 的"−34% 令牌"；真实数字约为 90%）。汇总**每轮助手 `usage`** 以获得真实总数。`total_cost_usd` 和 `duration_ms` 已经是累计/正确的。

复现：
```bash
bash scripts/agent-eval/bench-readme.sh      # 7 个仓库 × 有/无 × 4 次运行（RUNS=4）→ /tmp/ab-readme
node scripts/agent-eval/parse-bench-readme.mjs   # 中位数 + 节省百分比（汇总的每轮令牌）
```
