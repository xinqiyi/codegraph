# 直接回答 vs. 委托给 Explore 智能体（交互式 A/B）

**问题：** 在主会话中 *直接* 使用 CodeGraph 回答"某物如何工作"之类的问题，会使主会话上下文膨胀吗？——Claude Code 是否更适合将探索工作委托给一个可丢弃的 **Explore 智能体**（通过将文件读取吸收到子会话记录中来保持主上下文精简）？关键问题是：**这个答案会随规模变化吗**，在一个远比 Excalidraw 大的代码库上？

**简短回答：** 不会。使用 CodeGraph 时，主会话上下文大致是 **规模不变的（约 50k）**，因为检索是精准定向的，且 `explore` 的载荷受到预算限制——它不会在一个大 16 倍的仓库上膨胀。直接回答在 **每一种** 规模下都胜出：主上下文与委托路径一样或更精简，**零文件读取**，且令牌数减少约 28%。即使在大代码库上，委托带来的上下文精简优势也依然微乎其微。

## 方法论

- **测试工具：** 通过 `scripts/agent-eval/itrun.sh`（tmux）驱动的交互式 Claude Code TUI，**不是** 无头 `claude -p`。这一点很重要：无头模式会启动 **0** 个 Explore 智能体，因此完全无法衡量委托行为；只有交互式 TUI 才能做到。
- **实验组：** `WITH` = MCP 配置中有 CodeGraph；`WITHOUT` = 空的 MCP 配置（`--strict-mcp-config`）。
- **模型：** `opus`。**每个实验组 n = 3 次运行。** 主会话和子智能体的会话记录都被解析（`scripts/agent-eval/parse-session.mjs`）；读取和 bash 调用在主会话和子智能体之间累计。
- **仓库：** Excalidraw（643 个文件，中型）和 VS Code（约 10.7k 个文件，大型——约 Excalidraw 的 16 倍）。
- **构建版本：** 0.9.4。**日期：** 2026-05-24。
- "主会话上下文"是 TUI 报告的*主*线程的 `Context X/Y`（子智能体的上下文不计入）。"可计费令牌" = 每轮助手用量之和（输入 + 输出 + 缓存读取 + 缓存创建）。

## 发现

**使用 CodeGraph 时，主会话上下文是规模不变的。** 使用 codegraph 时，VS Code 上的主会话上下文约为 **47k——与 Excalidraw 的约 50k 基本一致**，尽管仓库大了 16 倍。它并没有膨胀。原因：codegraph 的 `explore` 载荷是 **预算受限的**，且检索是 **精准定向的**——回答一个问题只拉取相关的*流程/区域*，而不会仅仅因为仓库很大就拉取更多内容。因此 codegraph 使主会话上下文大致保持规模不变（约 50k）。即使在大代码库上，委托带来的上下文精简优势也依然微乎其微——与"在规模变大时变得显著"恰恰相反。

真正会随规模膨胀的是将许多大文件直接读取到主会话中——而 Claude Code **没有** codegraph 时通过委托给 Explore 智能体来避免这种情况（29–31k 主上下文），但代价是 **17–26 次读取** 和约 28% 的额外令牌。CodeGraph 以更好的方式保持主上下文精简：受限且精准的载荷——无需委托，**0 次读取**。

**关于"Explore 智能体也使用 codegraph"。** 我无法复现：在 **6/6** 次使用 codegraph 的运行中（两个仓库），Claude Code **从未委托**——它每次都直接回答。Explore 智能体路径仅出现在 `without` 实验组中（使用 grep/read，因为 codegraph 不在该配置中）。因此，在当前指令和 codegraph 存在的情况下，Claude Code 停留在主会话中——通过 Explore 智能体实现精简主上下文的最佳情况根本不会发生；实际情况是通过受限的 codegraph 实现精简主上下文，而且成本更低。

## 结论

**"使用 codegraph 直接回答"对 Claude Code 来说也是最佳选择——在任何规模下都是。** 不需要按智能体做分割；统一的"直接回答"指令对 Claude Code *以及* Codex / Cursor / opencode（它们没有 Explore 智能体机制，否则会直接读取文件）都是正确的。这一结论推动了 README 中 `## CodeGraph` 示例块的更新，该示例之前告诉智能体"永远不要直接调用 `codegraph_explore` / 始终生成一个 Explore 智能体"——也就是说，它引导 Claude Code 走向了*更差*（17–26 次读取，约 28% 更多令牌）的路径。

**注意事项 / 未来工作（非阻塞）：** 一个*本身使用 codegraph* 的 Explore 智能体原则上可以实现精简主上下文和低工作量两者兼得。但"直接回答"指令在实践中阻止了委托（6 次运行中观察到的委托数为 0），主上下文的收益将是微乎其微的（约 50k → 约 30k，都只占 1M 窗口的几个百分点），而且它增加了一次子智能体的往返。值得作为一个未来的实验，但不适合作为默认行为。
