# 动态分发覆盖指南

**受众：** 继续此工作的 Claude 代理。
**使命：** 系统地关闭 CodeGraph 支持的**每种语言和框架**中静态提取的**动态分发**覆盖漏洞，并以相同的方式验证每一个，使得跨符号的*流程*在图中的任何地方都存在。

> 这是顶层指南。一个机制（回调合成器）的深入设计在 [`callback-edge-synthesis.md`](./callback-edge-synthesis.md) 中。
> 完整调查上下文 + 发现：auto-memory `project_codegraph_read_displacement`。

---

## 1. 目标（为什么这很重要）

CodeGraph 的价值在于**成为地图**——回答 grep/Read 无法解决的结构/流程问题（`trace`、`impact`、调用者、"X 如何到达 Y"）。代理**只有在 codegraph 足够充分时**才会使用它而不是 Read。我们凭经验证明（参见 memory），充分性的杠杆是**覆盖率**，而不是提示/钩子/新工具：当流程在图**中缺失**时，代理会读取文件来重建它；当流程*在*图中时，代理可以在不读取的情况下完整回答问题。

**在 excalidraw 上端到端验证：** 在补全更新流程漏洞后，2/3 的无头代理运行以 **Read 0 和完整答案**回答了"更新如何到达屏幕"的问题——之前不可能，因为关键边不在图中。（注意：覆盖率*使能*了免读路径；代理确认性读取的方差意味着它并不*强制*如此。完整性无条件改善。）

使命是让这对**所有**语言/框架都成立。

## 2. 问题类别：动态分发

静态 tree-sitter 提取捕获显式调用（`foo()`、`this.bar()`）。它**遗漏**了任何目标是被计算/间接的调用。四种重复出现的形态，带有**难度梯度**（先做便宜的）：

| # | 形态 | 示例 | 修复机制 | 成本 |
|---|---|---|---|---|---|
| 1 | **命名属性 / 描述符** | django `self._iterable_class(self)` | 框架解析器（`claimsReference` + `resolve()`） | **便宜** |
| 2 | **字段支持的观察者** | `onUpdate(cb)` + `for(cb of cbs)cb()` | 回调合成器（全图遍历） | 中等 |
| 3 | **字符串键控的 EventEmitter** | `on('e',fn)` / `emit('e')` | 回调合成器（事件键控） | 中等 |
| 4 | **内联回调处理器** | `on('e', function h(){})` / `() => {}` | 提取（命名）+ 合成器链接通过主体（匿名） | 命名：便宜 · 匿名：困难 |

驱动机制选择的关键区别：
- **存在命名引用**可以解析（`_iterable_class` 是一个属性名）→ **解析器**。
- **不存在引用**（`cb()` 是匿名的；需要注册器↔分发器关联）→ **合成器**。

## 3. 实践示例（两种机制，端到端）

### 3a. Django ORM 描述符——**解析器**模式（Python）
- **漏洞：** `QuerySet._fetch_all` 调用 `self._iterable_class(self)`（一个运行时选择的迭代器，默认 `ModelIterable`），其 `__iter__` 运行 SQL 编译器。静态解析无法将属性解析为可调用对象 → `_fetch_all` 的唯一被调用者是 `_prefetch_related_objects`；`trace(_fetch_all, execute_sql)` 返回无路径。
- **修复：** `djangoResolver` 通过存在名称的预过滤器声明未解析的 `_iterable_class` 引用，然后将其解析为 `ModelIterable.__iter__`。
- **文件：** `src/resolution/types.ts`（`claimsReference?` 在 `FrameworkResolver` 上）、`src/resolution/index.ts`（`resolveOne` 中的预过滤器咨询 `claimsReference`）、`src/resolution/frameworks/python.ts`（`djangoResolver.resolve` + `claimsReference` + `resolveModelIterableIter`）。
- **结果：** `trace(_fetch_all, execute_sql)` → `_fetch_all → __iter__ → execute_sql`（3 跳）。

### 3b. Excalidraw 观察者 + EventEmitter——**合成器**（TS）
- **漏洞：** `Scene.triggerUpdate` 执行 `for (cb of this.callbacks) cb()`；`triggerRender` 通过 `scene.onUpdate(this.triggerRender)` 注册。`triggerUpdate → triggerRender` 边是动态的 → `trace` 返回无路径；整个更新流程断裂。
- **修复：** 一个全图遍历，检测注册器/分发器通道，关联注册站点，并合成 `dispatcher → callback` 边。再加上提取**命名的**内联回调，使得像 express 的 `function onmount(){}` 这样的处理器成为节点。
- **文件：** `src/resolution/callback-synthesizer.ts`（遍历——字段观察者 + EventEmitter）、`src/resolution/index.ts`（在 `resolveAndPersistBatched` 末尾调用 `synthesizeCallbackEdges()`）、`src/extraction/tree-sitter.ts`（`visitFunctionBody` 提取命名的嵌套函数）。
- **结果：** `trace(mutateElement, triggerRender)` → 3 跳；express `use → onmount`。

## 4. 可重复的方法论（每种语言/框架运行此流程）

### 步骤 1——选择框架的规范*流程*问题
每个框架都有一个标志性的数据/控制流。选择"X 如何到达/变成 Y"问题和一个真实的代码库（添加到 `.claude/skills/agent-eval/corpus.json`）。示例：
- React state→DOM，Vue reactive→render，Svelte store→update
- Rails request→controller→view，Spring request→`@Controller`→service
- Express/Koa request→middleware→handler，FastAPI request→route→dependency
- Redux action→reducer→store，RxJS subscribe→operator→observer
- 任何 ORM：查询构建器 → SQL 执行（django 模式）

### 步骤 2——测量漏洞（确定性，无代理）
```bash
rm -rf <repo>/.codegraph && ( cd <repo> && codegraph init -i )
node scripts/agent-eval/probe-trace.mjs <repo> <from-symbol> <to-symbol>   # 流程是否断裂？在哪里？
node scripts/agent-eval/probe-node.mjs  <repo> <break-symbol>              # 踪迹：下一跳是否缺失？
```
"没有直接调用路径…在动态分发处断裂" + 在断裂点处稀疏的踪迹 **定位了漏洞**（这正是发现 `_iterable_class` 和 `triggerUpdate` 的方式）。通过读取断裂符号的主体确认它是动态的。

### 步骤 3——分类 → 选择机制（使用第 2 节的表格）
- `self.<attr>(...)` / 描述符 / 元类 → **解析器**（第 3a 节）。
- `for(cb of store)cb()` / `store.forEach(cb => cb())` → **字段-观察者合成器**（第 3b 节）。
- `on('e',fn)` + `emit('e')` → **EventEmitter 合成器**（第 3b 节）。
- 内联处理器不是节点 → **命名的：** 提取（已在 `tree-sitter.ts` 中通用完成）；**匿名的：** 合成器链接通过主体（尚未构建）。

### 步骤 4——实现
- **解析器：** 添加到 `src/resolution/frameworks/<lang>.ts` ——一个 `resolve()` 分支 + `claimsReference(name)`，如果引用名称不是已声明的符号。复制 `djangoResolver`。
- **合成器通道：** 扩展 `src/resolution/callback-synthesizer.ts` ——添加框架的注册器/分发器**名称模式**和**主体模式**（例如信号使用 `.connect()`/`.emit()`；Rx 使用 `.subscribe()`/`.next()`）。
- 重新索引（步骤 2 命令）并重新运行 `probe-trace` ——流程现在应该连接起来。

### 步骤 5——验证（每次都相同的方式）
1. **确定性：** `probe-trace(from,to)` 找到路径；`probe-node` 显示已桥接的跳。之前断裂的跳已闭合。
2. **精度：** 计数 + 抽查合成/解析的边——没有爆炸，目标正确：
   ```bash
   sqlite3 <repo>/.codegraph/codegraph.db \
     "select s.name||' → '||t.name||'  '||coalesce(e.metadata,'') from edges e \
      join nodes s on e.source=s.id join nodes t on e.target=t.id where e.provenance='heuristic';"
   ```
   （解析器边不是 `heuristic`；改为通过 trace + callees 验证。）
3. **回归：** 节点计数稳定（前后执行 `select count(*) from nodes;` ——大幅跳转意味着提取更改过度触发）；控制代码库上的现有 trace 保持完整。
4. **端到端代理评估：** 使用 codegraph 运行流程问题并测量 **读取次数 / 答案完整性 / 成本** 与修复前基线的对比：
   ```bash
   # 无头模式（精确成本 + 干净的工具序列）
   bash scripts/agent-eval/run-agent.sh <repo> with "<flow question>"
   # 或完整的 A/B + 交互式探索子代理路径：
   scripts/agent-eval/audit.sh local <name> <url> "<flow question>" all
   ```
   然后解析：`Read` 计数、codegraph-tool 计数、成本，以及答案现在是否包含粘合符号（那些之前需要读取的符号）。

### 成功标准（每种语言/框架）
- `trace` 端到端找到规范流程（没有动态分发断裂）。
- 代理可以以 **Read 0**（在部分运行中可实现）回答流程问题，且粘合符号出现在答案中。
- **没有节点爆炸**，控制代码库上没有回归。
- 合成边在抽查时是精确的（没有通用名称过度链接）。

## 7. 已知限制与注意事项（来自 excalidraw/django 工作）

- **覆盖率使能，但不强制，免读路径。** 代理有时仍然读取以*确认*源代码；成本大致持平（codegraph 调用与读取交换）。可靠的胜利是**完整性** + 使 Read-0 *成为可能*。不要期望有保证的成本降低。
- **Vue（2026-05-23 验证，vitepress S / vben M / element-plus L）。** SFC `<template>` 未被提取器解析，因此模板使用需要合成（`vueTemplateEdges`）：`@click="fn"` → handler，kebab `<el-button>` → `ElButton`。PascalCase `<Child/>` 已经由 JSX 通道覆盖（SFC 组件节点跨越模板）。结果：代理读取在各种规模下都减少（vben login 1–3 对比 4–11），**在 handler 是本地函数时最强**（vben `handleLogin`/`handleSubmit`）。
  **可组合解构 handler 已解决：** `@click="closeSidebar"` 其中 `const { close: closeSidebar } = useSidebarControl()` 现在跟随别名 → 可组合 → 返回的 `close` 函数（当它在可组合的文件中定义时）。vitepress 边栏流程从 **6 → 0 次读取**下降（最佳情况）。仅精确——没有回退到可组合本身（静态的 `useX()` 调用边已经覆盖了那个），因此在返回的函数无法定位时什么都不添加（例如重新导出/外部可组合）。剩余限制：**前缀约定 kebab**——element-plus `el-button` → `button.vue`（组件名为 `button`，不是 `ElButton`），因此 kebab 在那里保持未解析；以及 **reactive→render**（vue-core Proxy 运行时）——深层框架内部前沿，推迟。
- **Svelte / SvelteKit（2026-05-23 验证，realworld S / skeleton M / shadcn L）——已经覆盖良好。**
  与 Vue 不同，`.svelte` 提取器已经解析模板：`extractTemplateCalls`（`{fn()}`）、`extractTemplateComponents`（`<Pascal/>` 组合——skeleton 956 / shadcn 1610 引用边），加上 `import * as api` 命名空间 + `load`→api 分辨率都有效。Agent A/B（realworld login）：使用 codegraph **1 次读取**对比不使用 **4 次**——codegraph 开箱即赢。一个提取差距是**函数对象**（`export const actions = { default: async () => {} }`；遍历器故意跳过对象字面量函数以避免内联对象噪音）。已为 **导出** 常量修复（一般性的——Redux/Express handler 映射也一样）；`extractFunction` `nameOverride` 保持内联对象箭头被跳过。**剩余：** 从提取的动作节点发出的 `$lib` 别名命名空间调用（`api.post`）即使相同的别名对 `load` 解析也不解析——一个更深的解析器交互，推迟（来自动作的本地/相对调用连接）。**教训：在假设漏洞之前先测量**——现代 Svelte 几乎不使用 `on:click={fn}`（而是使用表单动作/回调属性），因此假设的事件 handler 漏洞不是真正的那个；Svelte 需要的比 Vue 少得多。
